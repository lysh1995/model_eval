# Humanloop — Online Evaluators

Sources: https://humanloop.com/docs/explanation/evaluators, https://humanloop.com/docs/guides/observability/monitoring, https://humanloop.com/blog/llm-as-a-judge

## Online vs offline — defined by INPUT SHAPE, not by timing

This is Humanloop's conceptual contribution and it differs from every other platform surveyed.

**Online (monitoring)**, verbatim: "Evaluators are run against the **Logs generated by your AI applications**. Typically, they are used to **monitor deployed model performance over time and check for drift or degradation in performance**." The evaluator "receives **only the generated log as input**."

**Offline (evaluation)**, verbatim: works with "predefined datasets containing datapoints with inputs and optional target data. When you run an offline evaluation, **a Log needs to be generated using the inputs of each Datapoint** and the version of the application being evaluated."

The distinction: **online = no ground truth available (reference-free judging); offline = target/gold data exists.** Other platforms define online/offline by *when* it runs (production vs. CI). Humanloop defines it by *what the evaluator can see*. This is arguably the more fundamental cut — it dictates which metrics are even computable. In production you can never compute "correctness vs. gold answer"; you can only compute reference-free properties. This constrains the metric set far more than sampling does.

## Three evaluator sources (verbatim)

1. **Code** — "using simple deterministic rules based judgments against attributes like **cost, token usage, latency, regex rules on the output**, etc."
2. **AI** — "using other foundation models to provide judgments on the output. This allows for **more qualitative and nuanced judgments**"
3. **Human** — "getting gold standard judgments from either **end users** of your application, or **internal domain experts**"

Return types: Code and AI evaluators return **Boolean or Number**; Human evaluators return **Number, Select, Multi-select, or Text**.

Note the implied cost/fidelity ladder: code (free, deterministic) → AI (cheap, nuanced) → human (expensive, gold standard). This is the cascade axis expressed as *judgment source* rather than model size.

## Setup — auto-run on new logs

Verbatim steps:
1. "Go to your Prompt's dashboard. Click **Monitoring** in the top right to open the monitoring dialog."
2. "Click **Connect Evaluators** and select the Evaluator you created."
3. "Ensure the **'Auto-run' switch is enabled** for the Evaluator."

Verbatim: Humanloop "will **automatically run them on new Logs**." / "Online evaluators can be set to run automatically whenever logs are added to a project."

Re-run semantics, verbatim: "**Monitoring evaluators re-run on patched Logs and on Flow or Agent Logs when new Logs are added to their trace.**"

The re-run-on-patch behavior is a subtle cost multiplier: an evaluator can fire more than once per log if the log is amended, and agent/flow logs re-evaluate as their trace grows. **For a long multi-turn agent trace, evaluation cost is not O(1) per trace — it can be O(turns).** Combined with no sampling knob (below), this is a real cost exposure for conversational apps.

## Sampling — NOTABLE NEGATIVE FINDING

**No sampling rate configuration is documented** on either the Evaluators explanation page or the monitoring guide. The model is a binary **"Auto-run" toggle per evaluator**: on = every new Log, off = none.

Humanloop is the **only platform in this survey without a sampling rate primitive**. Cost control is expressed through evaluator *choice* (cheap code evaluators vs. expensive AI evaluators) and which Prompts you attach monitoring to — not through coverage percentage. Corroborates that the sampling knob is a convergent-but-not-universal design.

## The human-calibration loop — best-practice pattern worth stealing

Verbatim: "A best practice approach involves an **automated evaluator assessing all conversations, while a statistically significant sample is concurrently routed to human evaluators**. The results from both are **continuously compared to validate and calibrate the autorater's performance** and to identify novel issues, ensuring both long-term quality control and operational scalability."

This is a **two-tier design where sampling is applied to the HUMAN tier, not the machine tier**:
- Automated evaluator → **100% of conversations**
- Human evaluators → **statistically significant sample**
- Continuously compare the two → calibrate the autorater, detect novel failure modes

Reframes the whole sampling question. Everyone else samples to save *LLM* cost. Humanloop's framing: the truly scarce resource is **human attention**, so sample there — and the sample's purpose is not coverage but **measuring the machine evaluator's error against ground truth**. Connects directly to Arize's "start at 10-20% until you've validated your evaluator": both treat the sample as an instrument for trusting the autorater, rather than as a cost dial.
